# Feature Specification: Balance Module（余额模块）

**Feature Branch**: `001-balance-module`  
**Created**: 2026-03-07  
**Status**: Draft  
**Input**: User description: "创建余额模块，支持并发安全、防止超扣、单请求多交易、记录交易快照"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 并发安全余额更新 (Priority: P1)

[Describe this user journey in plain language]

作为业务调用方（例如支付网关或清结算服务），我可以在高并发场景下对同一账户持续发起余额变更请求（多笔扣款与加款），系统始终保持余额计算正确，不发生丢单、重复扣减或竞态条件导致的异常结果。

**Why this priority**:  
并发安全是金融场景中最核心的基础能力，直接影响资金正确性，一旦出错会带来重大损失，因此必须作为最高优先级。

**Independent Test**:  
通过构造高并发压力测试（如 1000 个并发请求，每个请求包含多笔金额变更），在已知初始余额和交易序列的前提下，验证最终余额与理论值严格一致，并且无异常错误或超扣发生。

**Acceptance Scenarios**:

1. **Given** 账户初始余额为 1000，**When** 并发 100 个请求每个扣款 5（顺序无关），**Then** 总扣款 500，最终余额为 500 且无错误返回。  
2. **Given** 账户初始余额为 0，**When** 并发多个请求尝试扣款任意正金额，**Then** 所有扣款请求都被拒绝或失败，最终余额仍为 0。  

---

### User Story 2 - 防止超扣与批量原子处理 (Priority: P1)

[Describe this user journey in plain language]

作为业务调用方，我可以在单个请求中提交多笔交易（包括扣款与加款），系统会按约定顺序依次处理，并保证在任一时刻余额不会低于允许阈值（例如 0）。如果某一笔交易会导致超扣，系统要么整体拒绝该批次，要么清晰地报告失败并确保最终余额满足约束。

**Why this priority**:  
防止超扣是资金安全基线，且单请求多交易是常见业务场景（如购物车结算、多订单同时支付），需要明确的原子性或可预期的失败语义。

**Independent Test**:  
通过单元与集成测试验证：给定一组交易序列，系统在任何处理顺序下都不会产生负余额；当某笔交易导致超扣时，本请求内的处理结果符合约定（例如全部回滚），并且返回结构可被上游明确识别和处理。

**Acceptance Scenarios**:

1. **Given** 账户余额为 100，**When** 单请求中提交三笔交易：扣 30、扣 40、扣 50，**Then** 系统整体拒绝该批次，最终余额仍为 100。  
2. **Given** 账户余额为 100，**When** 单请求中提交三笔交易：扣 30、扣 40、加 20，**Then** 三笔交易全部成功，最终余额为 50。  

---

### User Story 3 - 交易快照与审计追踪 (Priority: P2)

[Describe this user journey in plain language]

作为审计 / 风控 / 运维人员，我可以基于余额模块生成的交易快照，追踪任意一次余额变更的来源、明细以及前后状态，从而在出现争议或异常时快速定位问题。

**Why this priority**:  
资金系统必须具备可追溯性，快照记录是后续对账、审计、事后分析的关键基础，虽然不直接参与实时扣款，但对合规与问题分析非常重要。

**Independent Test**:  
通过集成测试在执行多笔交易后，查询对应的交易快照，验证快照中记录的账户标识、交易列表、前后余额、时间戳以及请求标识等信息与实际执行情况严格一致。

**Acceptance Scenarios**:

1. **Given** 一次成功的多笔交易请求，**When** 查询该请求对应的交易快照，**Then** 快照中包含请求 ID、账户 ID、交易列表（金额与方向）、前余额与后余额以及时间戳。  
2. **Given** 某请求因超扣被整体拒绝，**When** 查询快照，**Then** 快照中记录拒绝原因或失败状态，但不会记录影响账户余额的后余额变更。  

---

### Edge Cases

- 并发请求交叉读写同一账户余额时，最终结果仍满足不超扣约束。  
- 单请求交易列表为空时，系统视为无操作并返回成功，余额保持不变。  
- 单请求内存在重复幂等键的交易，系统只处理一次并返回一致结果。  
- 请求在处理中途失败（如网络中断）时，系统保证账户余额处于可预期的状态（例如保持请求前余额或明确的最终余额），并可通过快照复核。  

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 支持在单个请求中提交多笔交易（加款 / 扣款），并按定义顺序依次处理。  
- **FR-002**: 系统 MUST 在任意并发场景下保证账户余额计算正确，不出现丢单、重复扣减或错序导致的错误余额。  
- **FR-003**: 系统 MUST 在处理任意交易序列时确保余额不会低于允许阈值（例如 0），即防止超扣。  
- **FR-004**: 系统 MUST 在任意交易会导致超扣时，对该请求内的交易采用可配置的处理策略（例如全量回滚 / 全量拒绝），并返回清晰的失败原因。  
- **FR-005**: 系统 MUST 为每次余额变更生成交易快照，至少包含：账户标识、请求标识、交易列表（金额与方向）、前余额、后余额、时间戳。  
- **FR-006**: 系统 MUST 支持为请求指定幂等键，对同一幂等键的重复请求 MUST 产生相同结果且不会重复应用交易。  
- **FR-007**: 系统 MUST 提供查询接口，以请求标识或账户与时间范围为条件检索交易快照，用于审计和问题排查。  
- **FR-008**: 系统 MUST 将幂等相关信息持久化在专门的幂等记录表中，至少记录幂等键、请求内容哈希、请求处理状态、关联交易编号以及响应 / 错误数据，以支撑重复请求识别与结果重放。  
- **FR-009**: 系统 MUST 支持基于过期时间和创建时间对幂等记录进行安全清理，清理策略不得影响已落账交易的审计与可追溯性。  

### Key Entities *(include if feature involves data)*

- **AccountBalance（账户余额聚合）**:  
  表示某个账户当前可用余额及相关属性。最少包含：  
  - 账户 ID（唯一标识）  
  - 当前可用余额  
  - 币种  
  - 允许的最低余额（如 0）  
  - 版本号或序列号（用于检测并发更新冲突）  
  - 账户类型（如 CASH / MARGIN / FROZEN 等）  
  - 冻结余额（交易中 / 预授权）  
  - 总余额（可用余额 + 冻结余额，用于快速查询与对账）  
  - 是否允许负余额标记（如 allowNegative）  
  - 账户状态（如 ACTIVE / FROZEN / CLOSED）  
  - 创建时间与更新时间  
  任何对余额的变更 MUST 通过 BalanceTransaction 记录，禁止绕过交易直接修改余额。  

- **BalanceTransaction（交易流水表）**:  
  表示一次对账户余额的变更动作。最少包含：  
  - 交易 ID（唯一标识）  
  - 账户 ID  
  - 金额（使用可精确表示的数值，避免精度丢失）  
  - 方向（加 / 扣）  
  - 批次 ID（同一请求内多笔交易共享）  
  - 请求 ID（与上游请求关联）  
  - 幂等键（在账户维度内唯一，避免重复执行）  
  - 交易创建时间  
  - 业务交易编号（如 transaction_no，用于对外追踪）  
  - 币种  
  - 交易类型（如 DEPOSIT / WITHDRAW / TRANSFER / PAYMENT / REFUND）  
  - 交易前余额与交易后余额（用于独立重放与校验）  
  - 对手方账户及账户类型  
  - 业务参考号（如订单号 business_ref_no）  
  - 交易状态（如 PENDING / PROCESSING / SUCCESS / FAILED）  
  - 错误码与错误信息（失败原因）  
  - 锁类型（如 PESSIMISTIC / OPTIMISTIC）、重试次数与是否发生并发冲突标记；在当前实现中，锁类型默认使用 PESSIMISTIC，用于记录本次记账所采用的锁策略  
  - 业务时间与会计日期（分别表示交易发生时间与记账归属日期）  
  - 元数据（metadata，用于记录扩展业务信息）  
  - 客户端信息（client_info，用于风控与排障）  
  - 对账标记与对账完成时间（支持后置对账流程）  
  BalanceTransaction 记录为追加式（append-only），不得被物理删除或覆盖，以保证数据安全与可追溯。  

- **BalanceSnapshot（余额快照）**:  
  表示一次请求完成后账户余额的前后状态及其关联交易。最少包含：  
  - 快照 ID（唯一标识）  
  - 账户 ID  
  - 请求 ID  
  - 前余额  
  - 后余额  
  - 交易列表引用（指向该请求内的 BalanceTransaction 集合）  
  - 创建时间  
  - 状态（成功 / 拒绝）  
  - 账户类型  
  - 币种  
  - 会计日期（快照归属的会计日）  
  - 元数据（如该请求内交易编号列表或补充审计信息）  
  快照同样为追加式存储，用于审计和回放，不作为实时计算的唯一来源。  

- **IdempotencyRecord（幂等记录）**:  
  表示一次幂等调用的持久化记录。最少包含：  
  - 幂等键  
  - 请求内容哈希（用于区分相同键但参数不同的请求）  
  - 幂等处理状态（PROCESSING / SUCCESS / FAILED 等）  
  - 关联交易编号（如有）  
  - 响应数据快照  
  - 错误信息（失败场景）  
  - 过期时间  
  - 创建时间与更新时间  
  - 状态取值的枚举约定（至少包含 PROCESSING / SUCCESS / FAILED）  
  - 可扩展的元数据字段（如未来需要记录调用方标识等）  
  幂等记录用于防止重复执行同一业务请求，并支撑结果重放与清理策略设计。  

### Data Model Constraints for Concurrency & Safety

- 对于同一账户，在任意时刻只有一个写入路径可以修改 AccountBalance 的余额和版本号，具体通过数据层的原子性保证（如基于版本号的条件更新或等价机制），以防止并发写导致的丢失更新。  
- BalanceTransaction 与 BalanceSnapshot 为只追加实体，任何更正通过补充交易或快照完成，而不是直接修改历史记录。  
- BalanceTransaction 在账户 ID + 幂等键维度 MUST 唯一，用于防止同一业务请求被重复执行。  
- 任何跨账户的资金移动（如转账）必须以成对或成组的 BalanceTransaction 表示，并归属于同一个批次 ID，以便实现原子性和审计追踪。  
- 所有与账户标识相关的存储字段仅包含实现资金归集所需的最小信息，不包含与资金无关的敏感个人数据，以降低数据泄漏风险。  
- IdempotencyRecord 在幂等键 + 请求内容哈希维度 MUST 唯一，用于防止相同幂等键但参数不同的请求被错误视为同一次调用。  
- 在高并发账户和跨账户转账等高风险场景下，系统默认在数据库层使用行级悲观锁（如 `SELECT ... FOR UPDATE`）对相关账户记录加锁；对于涉及多个账户的操作，应用层 MUST 按固定顺序获取锁（例如按账户主键排序），以降低死锁风险并保证并发语义一致。  

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在模拟 1000 个并发请求、每个请求最多 10 笔交易的压力测试下，系统在无外部依赖瓶颈的环境中，所有请求处理完成后，账户实际余额与根据交易序列推导的理论余额完全一致，误差为 0。  
- **SC-002**: 在包含超扣场景的测试集中，系统对 100% 会导致余额低于阈值的请求进行拒绝或回滚处理，测试中不出现任何一笔负余额记录。  
- **SC-003**: 对同一幂等键重复发送的请求（至少 3 次），系统返回结果完全一致，且不会新增额外的交易记录或快照。  
- **SC-004**: 对于任意一次余额异常调查场景，审计人员在已知账户 ID 与时间范围的前提下，能够在 5 分钟内通过快照查询完整重放该时段内的余额变更过程。
